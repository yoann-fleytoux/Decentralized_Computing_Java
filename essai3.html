Atelier 1 : réalisation d'un serveur Web en Java

1

En repartant du travail fait en TD, on se propose de réaliser le serveur WEB. Ce TP se déroule pour partie sous windows pour la conception et la génération du schéma de classe, et sous Linux pour la partie programmation proprement dite. Toute la partie analyse se trouve dans « \\agate1\Public\JavRepartie\SystèmeRéparti-S4.uml »

Partie conception sous windows avec starUML


  1. Dès la connexion sous StarUML effectuée,

  • Importez le framework « java 1.3 » dans le paquetage « D- Analyse du domaine ». Actuellement, StartUML ne fournit pas le framework « java 1.5 », il faudra donc effectuer des ajustements directement dans le code généré ;

  • Inclure le profil « Java Profile ».


  1. Réalisez le diagramme de classes de conception détaillé.

  • Créez dans le paquetage « D- Analyse du domaine » les classes de conception ainsi que les interfaces ;

  • Déplacez dans le paquetage « D- Analyse du domaine », les classes métiers pertinentes. Utilisez les classes métiers lorsque cela est possible au lieu de créer de nouvelles classes dans le paquetage « D- Analyse du domaine » ;

  • Supprimez les associations qui ne vous semblent plus pertinentes. Attention, il ne faut en aucun cas réaliser une suppression définitive (« delete from model »). Il suffit d’utiliser la touche « suppr » ;

  • Positionnez la visibilité des rôles à « private » ;

  • Ajoutez les attributs ainsi que les méthodes des classes ;

  • Générez le diagramme de classes métier ainsi que le diagramme de classes de conception en sélectionnant les classes pertinentes ;

  • Ajoutez les paramètres de généricité lorsque cela est nécessaire ;

Attention, il faut rajouter les imports java nécessaires dans les différentes classes ;


  1. Faites une archive avec les classes générées en veillant à respecter la hiérarchie de fichiers et les répertoires qui les contiennent. On exclura de cette archive les paquetage Java qui sont inutiles car déjà inclus dans le langage Java.


  1. Transférez cette archive dans votre répertoire sous Linux avec l’outil FileZila.


Partie programmation sous Linux


  1. Extrayez les informations de l’archive dans le répertoire « Atelier1 » ;

Etape 1 : Compilation de l’application

L’objectif de cette étape est de compiler l’ensemble des classes générées par StarUML. Pour compiler les classes on rajoutera dans les méthodes retournant une valeur une instruction return avec une valeur « null » pour les objets, et une valeur par défaut pour les types de base.

Attention : StarUML ne faisant pas la différence entre une méthode et un constructeur, il faudra vérifier pour tous les constructeurs qu’ils ne sont pas précédés par « void ».

Etape 2 : Détection d’une demande

L’objectif de cette étape est de réaliser un serveur qui détecte une demande de connexion et retourne au navigateur la chaîne de caractères « Demande reconnue » suivie du flux reçu par le navigateur. Pour réaliser cette étape, il faudra modifier uniquement la classe « ServeurHTTP » et lui ajouter une méthode « main » utilisant un paramètre qui sera le numéro de port. Ce numéro vous sera donné par l’enseignant.

Etape 3 : Création d’un client

L’objectif de cette étape est de réaliser un serveur qui détecte une demande de connexion et crée le client pour lui répondre. Ce dernier retourne au navigateur la chaîne de caractères « demande reconnue ». Pour réaliser cette étape, il faudra modifier uniquement les classes « ServeurHTTP » et « Client ».

Etape 4 : Envoi d’un fichier texte au client

L’objectif est de compléter la classe « ChargeurFichierTexte » afin de pouvoir retourner au client un fichier Texte. Dans un premier temps, on testera cette classe en retournant le fichier « essai.html » qui devra se trouver dans le sous-répertoire « document » du répertoire « Atelier1 ».

Etape 5 : Analyser la demande de l’utilisateur

L’objectif de cette étape est de récupérer le flot de données de l’utilisateur, d’isoler le service demandé et dans le cas d’un formulaire d’extraire les différents champs et leur valeur. Pour simplifier, le service correspondra à un service lorsque le flux correspond à un formulaire et au fichier à charger lorsque le flux correspond au chargement d’un fichier.

Dans cette étape, il faut :

  • Développer l’interpréteur de flux provenant du navigateur. Pour simplifier, on considérera que tous les champs d’un formulaire ont un nom différent ;

  • Intégrer cet interprète au client pour que lorsque le serveur reçoit une demande, le client analyse la demande et fournit le résultat escompté.

Pour tester cette fonctionnalité vous chargerez le fichier « essai.html » et le formulaire « form1.html ».

Etape 6 : Traitement d’un formulaire

L’objectif de cette étape est le traitement d’un formulaire, c’est-à-dire la récupération des champs du formulaire leur interprétation et la mise en œuvre du service demandée. Dans notre cas le seul service fournit par le serveur est un service de calculatrice. Ce service permet d’effectuer l’opération sur deux opérandes. L’opération et les opérandes sont passées en paramètres. La réponse à une demande de calcul est le résultat du calcul dans un premier temps.

Etape 7 : Finalisation du serveur

Afin de finaliser le serveur, il faut intégrer les différentes classes développées lors des étapes précédentes afin d’obtenir un serveur http opérationnel. On veillera à ajouter une classe permettant de charger une image au format JPEG. On testera le serveur complet avec les fichiers « essai.html », « form1.html » et « essai3.html ».
